플랫폼팀 에이전트 상세 설계

데이터팀 산출물의 xAPI 이벤트 발화, postMessage 통신, OctoPlayer 라이프사이클, restore 복원을 Headless 브라우저로 자동 검증하는 에이전트. 실패 시 데이터팀에 이슈를 반려합니다.

검증영역 4 개
검증항목 24 개
테스트 Headless 자동
반려 대상 데이터팀
도구 Playwright
검증 아키텍처 HEADLESS

플랫폼팀은 OctoPlayer 뷰어를 모킹하여, 데이터팀 산출물(폴더)을 Headless 브라우저에서 로딩하고 4개 영역 24개 항목을 자동 검증합니다.

OctoPlayer 모킹 mock-viewer.html

실제 OctoPlayer 없이 테스트하기 위한 모킹 페이지입니다. 컨테이너+뷰어 역할을 하며, 콘텐츠 iframe과 postMessage를 주고받습니다.

🎭

테스트 파이프라인 5 STEPS

STEP 1. 테스트 환경 구성 셋업

Playwright 브라우저 인스턴스 생성 → mock-viewer.html 로딩 → 콘텐츠 폴더를 로컬 서버로 서빙 → iframe에 콘텐츠 로딩

STEP 2. 초기화 시퀀스 테스트 (P2+P3) 자동

requestUserId/requestRestoreData 캡처 → 자동 응답 → started 발화 검증 → iframeCurrentPage 캡처. 3초 타임아웃 적용.

STEP 3. 학습 시퀀스 테스트 (P1) 시뮬레이션

Playwright로 문항 인터랙션 시뮬레이션: 보기 클릭 → 제출 → completed 캡처 및 extensions 검증 → 해설 버튼 → viewed → 초기화 → reset. 문항 유형별 인터랙션 시나리오 자동 결정.

STEP 4. 종료 + 복원 테스트 (P3+P4) 시뮬레이션

terminated 전송 → left + deinit 검증 → 캡처된 restore로 새 iframe 로딩 → DOM 상태 복원 확인 → 재채점 일관성 검증

STEP 5. 리포트 생성 + 판정 출력

24개 항목 결과 집계 → PASSED/FAILED 판정 → platform_report_{id}.json 출력. 실패 시 데이터팀 반려.

📋 데이터팀에 전달되는 수정 요청
  • 수정 대상: octo-bridge.js (item.js는 수정 불가)
  • 이슈별 fix_guide: 구체적 함수명 + 수정 방향 포함
  • 재검증 범위: 수정된 이슈 관련 검증 항목만 재실행
  • 최대 반려: 2회. 3회차 실패 시 escalated 처리
⚠️ 반려 대상 구분: P1·P2·P4 이슈는 데이터팀(octo-bridge.js) 반려. 단, P3 라이프사이클 중 item.js 자체 오류가 원인인 경우 검수팀을 거쳐 제작팀으로 에스컬레이션합니다.
