검수팀 에이전트 상세 설계

제작팀 산출물(HTML 문항 폴더)을 4개 영역에서 전자동 LLM 검수하는 에이전트. 이슈 발견 시 위치+수정 가이드를 제작팀에 전달하여 부분 수정을 유도합니다.

검증방식 LLM 전자동
검증영역 4 개
검증항목 38 개
반려방식 부분 수정
최대반려 3 회
검수 아키텍처 FULL AUTO

검수팀은 제작팀 산출물(폴더)과 기획팀 메타정보(JSON)를 동시에 입력받아, 4개 영역 28개 항목을 자동 검증합니다.

🏗️

입력 → 검수 → 출력 구조

ARCHITECTURE ┌─────────────────────────────────────────────────────┐ │ 검수팀 입력 │ ├─────────────────────┬───────────────────────────────┤ │ 제작팀 산출물 │ 기획팀 메타정보 │ │ Q-{id}/ │ item_meta_{id}.json │ │ ├ index.html │ - item_type, difficulty │ │ ├ css/style.css │ - achievement_desc │ │ ├ js/item.js │ - content_element │ │ └ meta.json │ - authoring_guide │ └─────────────────────┴───────────────────────────────┘ │ ▼ ┌─────────────────────────────────────────────────────┐ │ 검수 엔진 (4개 영역 병렬 실행) │ │ │ │ [D1] 코드 무결성 → 규칙 기반 자동 검증 │ │ [D2] 교육적 정확성 → LLM 기반 콘텐츠 검수 │ │ [D3] 접근성·반응형 → 규칙 기반 자동 검증 │ │ [D4] 채점 로직 → 시뮬레이션 기반 테스트 │ └─────────────────────────────────────────────────────┘ │ ▼ ┌─────────────────────────────────────────────────────┐ │ 검수 리포트 review_report_{id}.json │ │ - status: "approved" | "rejected" │ │ - total_score: 0~100 │ │ - issues[]: 이슈 목록 + 수정 가이드 │ └─────────────────────────────────────────────────────┘
🔑 병렬 실행: D1(코드)과 D3(접근성)은 규칙 기반으로 ms 단위 완료, D2(교육)는 LLM 호출 필요, D4(채점)는 JS 시뮬레이션. 4개 영역을 병렬 실행하여 전체 검수 시간을 최소화합니다.
⚖️

판정 기준

판정조건후속 조치
✅ 승인 (APPROVED)4개 영역 모두 통과, Critical 이슈 0건, 총점 80점 이상데이터팀(TEAM 04)으로 전달
⚠️ 조건부 승인 (CONDITIONAL)Critical 0건이지만 Minor 이슈 존재, 총점 60~79점이슈 리스트와 함께 데이터팀 전달 (다음 버전에서 수정)
❌ 반려 (REJECTED)Critical 이슈 1건 이상 또는 총점 60점 미만이슈+수정가이드와 함께 제작팀으로 반려
📌 최대 반려 3회: 동일 문항이 3회 반려되면 해당 문항을 "status": "escalated"로 표시하고 수동 검토 큐로 이동합니다. 무한 반려 루프를 방지합니다.

각 영역별 검증 항목, 검증 방법, 심각도(Critical/Major/Minor)를 정의합니다.

D2. 교육적 정확성

LLM 기반 · 10항목
ID검증 항목검증 방법심각도

D2-02오답 타당성LLM이 각 오답 보기가 전형적 오개념에 기반하는지 평가Major

D2-05Bloom's 수준 부합LLM이 발문의 인지 수준을 독립 판단 → 메타 blooms_level과 일치 여부Minor
D2-06학년 수준 적합LLM이 어휘·문장 복잡도가 해당 학년에 적합한지 판단Major
D2-07해설 정확성LLM이 해설(피드백) 내용의 논리적 정확성 평가Major
D2-08문항 독립성문항이 외부 맥락 없이 단독 풀이 가능한지 확인Minor
D2-09메타편집 재생성 정합성사용자가 메타정보를 수정하여 재생성한 경우, 수정된 메타(난이도, 문항유형, Bloom's 등)와 실제 문항 콘텐츠가 일치하는지 LLM 교차검증. 예: 난이도 "상"인데 문항이 단순 계산인 경우 부적합Critical
D2-10수식 정확성 (수식 문항)MathType 수식 입력이 필요한 문항에서 제시된 수식과 정답의 수학적 정확성을 LLM이 독립 검증Critical
🤖 LLM 교차 검증: D2 검증에는 제작팀과 다른 모델 또는 다른 온도(temperature)를 사용합니다. 동일 모델·동일 설정이면 같은 오류를 반복할 위험이 있으므로, 검수 LLM은 temperature: 0.1로 고정하여 일관된 판단을 합니다.
📌 D2-09 메타편집 재생성 검증: 사용자가 드롭다운으로 메타정보를 수정(예: 문항유형 "객관식"→"서술형", 난이도 "하"→"상")한 뒤 재생성된 문항은, 수정된 메타 JSON과 실제 문항 콘텐츠 사이의 정합성을 LLM이 5개 축(유형 일치, 난이도 적절, Bloom's 부합, 배점 적합, 영역 부합)으로 검증합니다.

검수 파이프라인 5 STEPS

4개 영역을 병렬 실행하고, 결과를 합산하여 판정합니다.

STEP 1. 입력 수신 및 전처리 전처리

제작팀 폴더와 기획팀 meta.json을 수신합니다. 파일 존재 여부를 먼저 확인하고(D1-02), 실패 시 즉시 Critical 반려합니다.

STEP 2. 4개 영역 병렬 검증 병렬

D1~D4를 동시에 실행합니다. 각 영역은 독립적으로 검증하고 결과를 반환합니다.

D1 코드 무결성 ~50ms

  • HTML/CSS/JS 파싱
  • 필수 파일·함수 확인
  • 외부 리소스 스캔

D2 교육적 정확성 ~3s

  • LLM 직접 풀이 + 정답 비교
  • 성취기준·난이도 부합 판단
  • 어휘 수준 평가

D3 접근성·반응형 ~100ms

  • ARIA 속성 스캔
  • 키보드 핸들러 확인
  • 미디어쿼리·폰트 검증

D4 채점 로직 ~2s

  • 정답/오답/미응답 시뮬레이션
  • getResult() 구조 검증
  • 타이머 동작 확인

STEP 3. 점수 산출 산출

각 영역의 검증 결과를 가중합산하여 총점(0~100)을 산출합니다.

D1 코드 무결성
25점 (Critical 항목 실패 시 0점 고정)
D2 교육적 정확성
30점 (메타편집 정합성 포함)
D3 접근성·반응형
20점 (UDL v0.1 반영으로 비중 상향)
D4 채점 로직
25점 (Critical 항목 실패 시 0점 고정)
0 (반려)60 (조건부)80 (승인)100

STEP 4. 판정 판정

총점 + Critical 이슈 유무로 최종 판정합니다. Critical 이슈가 1건이라도 있으면 총점 무관하게 반려.

STEP 5. 리포트 출력 출력

검수 리포트 JSON을 생성하여 출력합니다. 반려 시 이슈별 위치+수정 가이드를 포함합니다.

검수 리포트 JSON

검수 결과를 구조화된 JSON으로 출력합니다. 승인/반려 판정 + 영역별 점수 + 이슈 목록 + 수정 가이드를 포함합니다.
